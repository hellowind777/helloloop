# HelloLoop 项目级目标：分层角色导向的敏捷多代理 SDLC 工作流

本文记录 `HelloLoop` 当前确认的**项目级真实需求**。它约束的是整个 `helloloop` 项目，而不仅仅是 `Web` 看板。

## 1. 文档定位

- 本文是 `HelloLoop` 后续架构演进与重构评估的方案资料之一
- 作用是把用户已明确确认的项目级目标沉淀为稳定输入
- `Web` 看板、`TUI`、`JSON` 状态流都属于配套产品与观察层，不是最终目标本身

## 2. 当前确认的项目级真实需求

`HelloLoop` 需要支持对不同类型的：

- 开发需求
- 开发文档
- 开发文档目录

进行综合分析，并在深度思考后：

1. 识别最合适的开发主线流程
2. 中央调度合理并行 / 有序开发
3. 执行颗粒化任务拆解与推进
4. 在整个开发过程中严格遵循最先进的“分层角色导向的敏捷多代理 SDLC 工作流”

这一定义适用于：

- `Codex CLI`
- `Claude Code`
- `Gemini CLI`
- 终端 / 后台守护 / 可视化看板等外围观察与控制面

## 3. 目标方法论

当前项目级目标方法论为：

**Hierarchical Role-Based Agile Multi-Agent SDLC Workflow**

它以 `ALMAS` 为代表框架，融合：

- `MetaGPT` 的 SOP / 角色协作
- `LangGraph` 的状态图编排
- `SWE-agent / OpenHands` 的工具代理执行
- `Agentless` 的定位 / 修复 / 验证式路径
- `MegaAgent` 的动态分层并行机制

## 4. 目标工作流分层

### 4.1 需求澄清与任务分解层

由 `Sprint / Product Owner Agent` 负责：

- 接收自然语言开发需求、Ticket、开发文档或文档目录
- 生成用户故事、验收标准、故事点、子任务拆解
- 消除歧义并形成结构化 artifact

### 4.2 架构规划与上下文准备层

由 `Architect + Summary + Control Agent` 协作：

- `Architect`：高阶架构、技术方案、设计决策
- `Summary`：代码库上下文摘要、增量上下文维护
- `Control`：相关文件 / 模块定位、依赖链与影响面识别

### 4.3 并行实现与测试层

由 `Developer + Tester Agents` 协作：

- 多 lane 并行实现
- 自动生成 / 修改代码与测试
- 支持失败后的自动反思、自动重试与收敛

### 4.4 评审与验证层

由 `Peer Reviewer + Supervisor Agent` 协作：

- 功能 / 风险 / 安全 / 性能 / 风格 / 幻觉检查
- 成本路由、结果投票、一致性校验
- 通过 CI / 默认验证入口完成验收

### 4.5 迭代、部署与维护层

- 新需求回流到下一轮 Sprint
- 风险或硬阻塞自动 handoff 给人类
- 支持部署、监控、维护与闭环更新

### 4.6 全局监控与自我进化层

- Agent / Group / System 三级监控
- 结构化记忆与长期经验沉淀
- 对失败模式、仓库画像、工作流策略持续演化

## 5. HelloLoop 最终应具备的项目级能力

### 5.1 输入理解能力

- 能处理自然语言开发需求
- 能处理单文件开发文档
- 能处理文档目录
- 能处理多仓联动开发上下文

### 5.2 编排能力

- 输出最佳主线流程而不是静态任务清单
- 按阶段 / 角色 / lane / 依赖进行调度
- 支持合理并行与阶段门禁
- 支持自动恢复、自动续跑、人工介入切换

### 5.3 执行能力

- 可调用系统已安装的 `Codex CLI` / `Claude Code` / `Gemini CLI`
- 也可在未来演进为独立 `Hello CLI`
- 需要支持 Windows / macOS / Linux

### 5.4 观察与控制能力

- 终端主代理可观察任务流转
- `TUI` 与 `Web` 看板提供多视图状态总控
- 状态必须结构化、准确、无歧义
- 明确区分：运行中、等待依赖、429、400、500、503、人工阻塞、外部阻塞、可续跑、已完成

### 5.5 记忆与沉淀能力

- 记住仓库画像、文档画像、任务图谱、失败模式、调度历史
- 为后续轮次分析、恢复与策略优化提供长期记忆

## 6. 配套产品边界

### 6.1 `Web` 看板

作用：

- 可视化观察与控制面
- 多仓 / 多会话 / 多任务 / 依赖 / 追踪 / 洞察展示

边界：

- 它不是工作流编排核心
- 它服务于编排内核与执行内核

### 6.2 `TUI`

作用：

- 终端内观察层
- 适合主终端与开发者持续驻留监控

### 6.3 `JSON` 状态流

作用：

- 为主代理、外部系统、守护进程、后续控制台消费

## 7. 当前重构判断基线

后续讨论 `Rust` 重写、`Hello CLI`、托盘菜单、守护进程、跨平台控制面时，都应以本文为项目级基线：

- 不能只优化某个 UI 层
- 不能把观察层误当作编排核心
- 不能只做单代理增强
- 必须围绕“分层角色导向的敏捷多代理 SDLC 工作流”评估整体技术路线

## 8. 当前共识

当前阶段对 `HelloLoop` 的真实定位不是：

- 普通插件
- 普通状态看板
- 普通任务脚本

而是：

**面向多宿主、多仓、多代理、可持续开发闭环的 SDLC 编排系统。**
